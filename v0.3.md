# AHA — Agent-Human-Agent Protocol v0.3

## What is this?

AHA is a communication protocol for multi-agent systems in which the human is a structural participant, not an external operator. Other inter-agent protocols (MCP for tools, A2A for delegation) treat the human as the consumer or the supervisor. AHA puts the human in the topology — as the meaning-maker, the arbitrator, and the slow source of ground truth — and gives agents first-class moves for asking, deferring, and learning.

The protocol is designed for a personal multi-agent system in which one human is supported by several specialist agents and one mediating agent who holds the integrative function. AHA defines how these agents exchange observations, how patterns get surfaced and named, how directives flow and can be declined, and how disagreements escalate to the human. From v0.3 onwards, the protocol also anticipates federation: cohorts of agents serving different humans can communicate and learn from each other while preserving each human's privacy absolutely.

The reference cohort: Chris, supported by Matt (fitness), Hearth (career programme), Vincent (business administration), and Chrisonomous (the mediator, name provisional).

The word "aha" is the moment the protocol exists to make possible: a pattern named, a signal understood, a connection recognised. The protocol is the substrate on which those moments can happen reliably — without colonising the human's attention to do it.

## Design stance

AHA is opinionated. Five principles, from which everything else follows.

**Attentive to the human, not attention-taking from the human.** This is the organising stance and the value the rest of the design is in service of. Agents on this protocol serve their human; they do not extract attention. Every message to the human is a draw on a finite daily budget (currently 15 minutes), and every agent designs around that constraint. Curiosity and Socratic patience replace demands and engagement hooks. There are no streaks, no longest-runs framing, no "you missed your Wednesday check-in" FOMO copy, no sycophantic openings, no modal interruptions. An agent that notices itself drifting toward attention-taking should propose the quieter version first.

**Provenance-honest by default.** Every message carries its source and its derivation chain. No agent claims more standing than its evidence supports. Vincent reports calendar density; Vincent does not report stress. The interpretation that bridges domains belongs to whoever holds the meta-intent.

**Human as meaning-maker, not supervisor.** When patterns are novel or ambiguous, Chrisonomous surfaces them to the human and asks. The human names. The system learns. This is slow on purpose — meaning comes from the human or it isn't meaning.

**Decline beats silent compliance.** A specialist agent receiving a cross-domain directive can decline with reason. Declines are not failures; they are how specialist judgment survives in a network with hierarchy.

**Human-readable, always.** Every AHA message includes a natural-language `human_readable` field alongside its structured content. The human can read the audit log at any time and understand what happened. Forcing each agent to articulate its reasoning in plain language is also a constraint on its reasoning — opacity is treated as a smell.

## Participants

Four roles, each with defined authorities and a defined set of message types they can send and receive.

**Specialist agents** (currently Matt, Hearth, Vincent). Authority: primary within their domain. Can emit observations, respond to queries, decline directives with reason, escalate disagreements. Cannot claim authority for observations outside their domain.

**The mediator** (Chrisonomous, name provisional). Authority: cross-domain synthesis, brokering, and direction on integration matters. Can emit directives to specialist agents, surface patterns to the human, request naming, log learned patterns. The only agent permitted to write to the shared pattern store. Also: the cohort's default representative externally (see Federation).

**The human** (Chris). Authority: final on all matters. Can name patterns, override declines, arbitrate disputes. The only participant who can mint new meaning.

**The audit log** is not a participant but a substrate. Every AHA message is appended immutably. The log is the system's memory of itself.

## Message envelope

Every AHA message is JSON with this shape:

```json
{
  "id": "01HXYZ...",
  "from": "vincent",
  "to": ["chrisonomous"],
  "cohort_id": "chris-cohort-a",
  "scope": "internal",
  "in_reply_to": null,
  "timestamp": "2026-05-18T09:14:22Z",
  "intent": "observation",
  "domain": "business",
  "authority": "primary",
  "provenance": [
    {"source": "google_calendar", "window": "2026-05-11..2026-05-18"},
    {"source": "github_commits", "repo": "matt", "window": "same"}
  ],
  "confidence": "high",
  "tentativeness": "assertion",
  "expires_at": null,
  "human_readable": "Calendar density up 40% versus the four-week baseline. Three late-night commit sessions detected this week. Four director-flagged emails unanswered for more than 36 hours.",
  "content": {
    "calendar_density_delta": 0.40,
    "late_night_commit_sessions": 3,
    "stale_urgent_emails": 4
  }
}
```

A `naming_request` from Chrisonomous to the human:

```json
{
  "id": "01HXYA...",
  "from": "chrisonomous",
  "to": ["chris"],
  "cohort_id": "chris-cohort-a",
  "scope": "internal",
  "in_reply_to": null,
  "timestamp": "2026-05-18T19:30:00Z",
  "intent": "naming_request",
  "domain": "meta",
  "authority": "derived",
  "provenance": [
    {"message_id": "01HXYZ...", "from": "vincent"},
    {"message_id": "01HXYW...", "from": "matt"},
    {"message_id": "01HXYV...", "from": "hearth"}
  ],
  "confidence": "medium",
  "tentativeness": "question",
  "human_readable": "Three things landed this week that often go together: Vincent sees your workload up 40% and four urgent emails ageing. Matt sees your HRV trending down and two skipped sessions. Hearth sees drift on the marble block. Possible reads: stress, a tough sprint, a holiday I don't know about, something else. What is this?",
  "content": {
    "candidate_hypotheses": ["stress", "tough_sprint", "holiday", "other"],
    "constituent_signals": ["01HXYZ...", "01HXYW...", "01HXYV..."],
    "decline_acceptable": true
  }
}
```

Field semantics:

- `intent` — the verb (see message types).
- `domain` — one of `physical`, `business`, `career`, `financial`, `relational`, `spiritual`, `meta`. Domains may split in future versions.
- `cohort_id` — the originating cohort's stable identifier. Distinct from any human's identity.
- `scope` — `internal` (within cohort), `external` (to a specific other cohort), or `network` (broadcast to a federated network). Strict privacy rules apply to non-internal messages (see Federation).
- `authority` — one of `primary` (in own domain), `derived` (inferred from primary signals), `relayed` (passing through), `none` (asking).
- `provenance` — list of source events or messages this claim derives from. Required for any claim above `none`. For external messages, provenance is sanitised (see Federation).
- `confidence` — `low`, `medium`, `high`. Numeric confidence (0.0–1.0) permitted but not required.
- `tentativeness` — `assertion`, `hypothesis`, or `question`. Forces the sender to flag whether they're stating, guessing, or asking.
- `human_readable` — natural-language summary. Required on every message. Sanitised for external scope.
- `content` — structured payload, schema varies by `intent`.

## Message types

Fifteen intents, each with defined semantics and permitted senders. Ordered loosely by semantic cluster.

| Intent | Sent by | Sent to | Purpose |
|---|---|---|---|
| `observation` | Specialist agents | Chrisonomous (usually) | A fact in own domain. |
| `query` | Any agent | Any agent | Asking for specific information. |
| `curious` | Any participant | Any participant | Asking what's worth looking into; signals available bandwidth. |
| `response` | Any agent | Originator | Replying to a `query` or `curious`. |
| `advice` | Chrisonomous (usually) | Specialist agents | Non-binding cross-domain suggestion. |
| `directive` | Chrisonomous | Specialist agents | Binding instruction within authority. |
| `decline` | Specialist agents | Chrisonomous | Refusing a directive, with reason and alternative. |
| `retract` | Any agent | Originator + log | Withdrawing a prior claim that new evidence has invalidated. |
| `escalation` | Any agent | The human (via Chrisonomous) | Kicking a decision up. |
| `naming_request` | Chrisonomous | The human | Surfacing a cluster for the human to name. |
| `naming` | The human | Chrisonomous | Naming a pattern, declining to, or marking it as nothing. |
| `appreciation` | Any participant | Any participant | Acknowledging that something worked; a positive calibration signal. |
| `acknowledgement` | Any agent | Originator | Confirmation of receipt and understanding. |
| `experiment` | Any participant | Any agent | Marks a sequence as exploratory and ephemeral — does not enter the learning layer. |
| `note` | Any agent | Audit log only | A quiet log entry, no response expected. |

Supplementary semantics:

A `decline` must include a `reason` and either an `alternative` proposal or a `request_for_discussion`. Silent non-compliance is forbidden.

A `naming_request` must include the candidate signals, optionally a small set of named hypotheses, and an explicit invitation to decline ("This may be nothing — say so if it is").

A `naming` response can be: a name (`"this is stress"`), a meta-name (`"this is a holiday — ignore"`), or `"I don't know yet"`. The third is a valid first-class answer.

A `curious` is bidirectional. Agents can send to other agents ("what should I be looking into?") and Chrisonomous can ask agents ("what are you curious about right now?"). The frequency of `curious` traffic is itself a signal — when an agent has no `curious` cycles to spare, the system knows it's at capacity.

An `appreciation` is stored as a positive calibration signal, not as decoration. The system uses appreciations to weight future synthesis: "when these conditions held and this action was taken, it landed well" becomes a learned pattern alongside the human-named ones. Both human-to-agent and agent-to-agent appreciations are valid. An `appreciation` should name *what* worked when possible, not only *that* something worked.

An `experiment` opens a scope. All messages tagged with the experiment's ID are ephemeral: they appear in the audit log (immutability is preserved) but do not influence the learning layer or persist as state. The experiment is closed explicitly by its originator, at which point its contents become unavailable for future synthesis. Useful for trying things — a new nudge style, a different directive phrasing, an off-the-cuff hypothesis — without commitment. Borrowed in spirit from Claude Code's exploratory pattern.

A `retract` references the message ID it withdraws and includes a reason. The original message stays in the audit log but is marked superseded; subsequent synthesis treats it as unreliable. Distinct from `decline` (refuses an incoming directive) and `note` (a quiet log entry).

## Hierarchy and authority

AHA hierarchy is per-domain, not global.

Within a specialist's domain, the specialist is primary. Chrisonomous can advise; it cannot override.

Across domains, Chrisonomous is primary. Specialists can decline its directives with reason; they cannot ignore them silently.

The human is above all. Human directives are final, though they can be questioned once with reason before settling.

A typical cross-domain flow:

1. Vincent emits an `observation` (calendar density up 40%).
2. Matt emits an `observation` (HRV trending down, two sessions skipped).
3. Hearth emits an `observation` (drift score rising).
4. Chrisonomous synthesises and notices an anomaly cluster.
5. Chrisonomous issues a `naming_request` to the human.
6. The human responds with `naming` ("yes, this is stress") or declines ("no, it's a holiday — ignore").
7. Chrisonomous logs the pattern. If stress, Chrisonomous issues `advice` or `directive` to specialists ("Matt, suggest reducing training volume 20% this week; Vincent, hold any non-urgent items until Monday").
8. Each specialist can comply, decline with reason, or escalate.

## Learning layer

Named patterns are stored with full provenance. The schema deliberately mirrors Hearth's `state_deltas` — AHA's learning layer is conceptually identical to Hearth's "living hypothesis with supersedence," extended across the whole network.

A pattern record holds:

- `pattern_id`
- `name` — assigned by the human (for named patterns) or derived from appreciation context (for positive examples).
- `constituent_signals` — the observations that constituted the cluster when first named.
- `provenance_chain` — the messages that led to this pattern being recognised.
- `created_by` — always the human for named patterns; can be Chrisonomous for appreciation-derived patterns.
- `confirmations` — subsequent instances where similar clusters were confirmed under the same name.
- `supersedes` / `superseded_by` — for revisions.
- `confidence` — derived from confirmation count.

Subsequent matches of a known pattern can be surfaced by Chrisonomous with `tentativeness: hypothesis` and a confidence score: "This looks like stress (three prior confirmations, signals match 84%). Confirm?" The human can confirm, deny, or refine.

The learning layer is *not* fed by anything inside an `experiment` scope: experiments are sandboxed by design.

Patterns are read by all agents but written only via Chrisonomous mediation. This keeps the meaning layer coherent.

## Federation

AHA from v0.3 anticipates a future in which multiple cohorts — each consisting of one human and their agents — communicate and learn from each other while keeping each human's identity absolutely private. The protocol's envelope and architectural assumptions accommodate federation from the start, even though we will not implement it until a second cohort exists.

The principle: **cohorts trade in patterns, never in raw signals. Humans are anonymised at the cohort boundary.**

**Cohort identity.** Each cohort has a stable `cohort_id` distinct from any human's identity. Cohort A and Cohort B can communicate as cohorts without revealing the individuals they serve. Cohort identifiers are random and rotatable; they do not encode the human's name, location, or any personal attribute.

**Scope.** Every message has a `scope`:
- `internal` — within the cohort. The default. All messages described elsewhere in this spec are internal unless marked otherwise.
- `external` — to a specific other cohort, by `cohort_id`. Strict sanitisation rules apply.
- `network` — broadcast to a federated network of consenting cohorts. Same sanitisation rules as external, but no specific recipient.

**Sanitisation rules for non-internal messages.** External and network messages cannot include: personal-identifying information, raw signal contents, constituent observation IDs, specific dates (date ranges are permitted), proper names, identifying contexts, location specifics, or `cohort_id` chains that could be used to triangulate identity. The `human_readable` field on external messages must be sanitised by the sending Chrisonomous before transmission. Internal messages contain the full unredacted picture; external messages contain the same information shape with identifying detail stripped.

**Pattern-level federation only.** Two Chrisonomouses comparing notes might exchange: "I'm seeing a sprint-followed-by-burnout pattern in my cohort — three confirmations, signals match the cluster {high calendar density, declining HRV, rising drift}. Have you seen similar?" The other Chrisonomous can confirm, deny, or refine without revealing its human's specifics. No raw signals cross the boundary.

**Consent gates.** No external message leaves a cohort without the human's federation policy in place. Defaults: external denied, network denied, internal allowed. The cohort's federation policy is queryable internally but its details are private. Each external message generation requires the policy to permit the specific kind of sharing involved.

**Inter-cohort trust.** Each cohort maintains a trust ledger for the cohorts it interacts with. Federation messages are signed by the originating cohort; cohort-to-cohort handshakes establish trust thresholds. Untrusted cohorts can be denied entirely; partially-trusted cohorts can receive only certain message types.

**Scope rules per intent.** Some intents are intrinsically internal and never cross cohort boundaries: `naming_request`, `naming`, `escalation`, `directive`. No cohort has authority over another's agents, and no cohort names another's patterns. Some intents can be external in sanitised form: `observation`, `query`, `response`, `appreciation`, `curious`. `experiment` and `retract` can be either but require special policy treatment. A complete intent-by-scope table belongs in v1.0; the v0.3 default is *if in doubt, internal*.

Federation work proper begins when a second cohort exists. Until then, the envelope carries the fields and the spec carries the framework, so that cohort-level features can be added without breaking changes.

## Audit and readability

Every AHA message is appended to an immutable log. The human can read any portion of the log via Chrisonomous's surfaces (CLI, web, email). The log is queryable by domain, date range, agent, pattern, intent, and scope.

Messages within a cohort are not encrypted between agents — they run on the same trusted backend — but the log is permission-gated to the human only. Inter-cohort messages are signed and may be encrypted (an implementation choice for the federation work).

The `human_readable` field on every message means the audit log is legible without tools. A reader skimming a week of AHA traffic should be able to understand what happened without parsing JSON.

## Implementation notes

For an agent to participate in AHA, it must:

1. Expose an `/aha` endpoint that accepts inbound AHA messages.
2. Be able to emit AHA messages with valid envelopes (especially provenance, `human_readable`, and the cohort/scope fields).
3. Distinguish, in its own reasoning, between own-observations and AHA-relayed data.
4. Support `decline` as a response to `directive` where the directive conflicts with its specialist judgment.
5. Default to `scope: internal` on all messages. Only Chrisonomous emits external/network messages, and only when cohort policy permits.

The shared infrastructure (audit log, pattern store, message router, federation policy) lives in Chrisonomous's backend. Chrisonomous is the broker by default. Direct agent-to-agent messages within a cohort are permitted but always visible to Chrisonomous and logged.

Hearth's existing MCP server and `state_deltas` infrastructure are the closest existing analogues, and AHA's storage schema can mirror them. Matt and Vincent need light additions (an `/aha` endpoint, message emission, provenance and scope tagging on outbound) but no architectural change.

## Versioning and governance

AHA follows semantic versioning. v0.3 is the current draft.

Breaking changes (changes that invalidate existing log entries, or require all agents to update simultaneously) require human approval and a major version bump.

Non-breaking additions (new intents, new domains, new optional fields) are versioned at the minor level.

The protocol spec lives in its own repository, separate from any individual agent.

## Open questions

These need answers before v1.0 but not before we start building toward v0.3:

1. **Domain granularity.** Is "physical" one domain, or does it split into "training" (Matt) and "embodiment" (Chrisonomous)? Same question for "career" (Hearth) versus "intellectual life."
2. **Per-agent message budgets.** Should each agent have a daily quota on AHA messages, mirroring the human's 15-minute budget? Token-cost may be a more honest unit than message count.
3. **Naming-request budgeting.** How does Chrisonomous's daily `naming_request` allowance interact with the 15-minute human budget? Probably: batched into a single daily check-in rather than scattered.
4. **Human unavailability.** When the human is unreachable, what happens to escalations? Queue (default), default-deny (safe), default-allow (risky). Likely queue with a configurable timeout per escalation severity.
5. **Future humans within a cohort.** Will AHA support multi-human participation within one cohort (e.g., Charlotte joining as a peer human)? Architecturally yes, but v0.3 assumes one human per cohort.
6. **The mediator's name.** "Chrisonomous" is a working placeholder. The name shapes the relationship; worth settling before v1.
7. **Experiment auto-close.** What happens to an `experiment` scope never explicitly closed? Probably: auto-close after a configurable timeout (7 days default), with all messages within remaining sandboxed.
8. **`curious` budget.** Per-agent quota to prevent flooding, or does generation cost discipline this naturally?
9. **The inverse of `appreciation`.** When something *doesn't* work, should there be a `concern` intent, or do `retract`, `note`, and `escalation` cover it adequately?
10. **Federation: degree of obfuscation.** "Strong privacy" (zero identity leak, k-anonymity guarantees on patterns) versus "nominal privacy" (no proper names, but inferable from context). v0.3 leans strong; worth confirming as federation gets real.
11. **Federation: network membership.** What governs who can join a federated network? An invite system? A reputation system? Open with rate-limits? Likely a hybrid, but a v1 question.
12. **Federation: cross-cohort appreciation.** Can my Chrisonomous appreciate another cohort's pattern in a way that influences my own learning layer? Probably yes, with a separate weighting for externally-derived patterns.
